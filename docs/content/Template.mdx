## Examples

### Simple example

```jsx live
<ActionMenu
  anchorContent="Menu"
  onAction={item => console.log(item)}
  items={[
    {text: 'New file', key: 'new-file'},
    {text: 'Copy link', key: 'copy-link'},
    {text: 'Edit file', key: 'edit-file'}
  ]}
/>
```

### Example with Item states

Action list items can be selected. Single selections are represented with a check octicon, while multiple selections are represented with a checkbox component. These selection visuals are always placed at the beginning of the item.

An action list item can have a special "danger" style, to be used in cases that require extra attention from the user. Place danger items at the end of the list.

```jsx live
<ActionMenu
  anchorContent="Menu"
  onAction={item => console.log(item)}
  items={[
    {text: 'Default', key: 'default'},
    {text: 'Disabled', disabled: true, key: 'disabled'},
    {text: 'Danger', variant: 'danger', key: 'danger'},
    {text: 'Selected', selected: true, key: 'selected'}
  ]}
/>
```

### Example with Divider

Item dividers allow users to parse heavier amounts of information. They're placed between items and are useful in complex lists, particularly when descriptions or multi-line text is present.

```jsx live
<ActionMenu
  anchorContent="Menu"
  onAction={item => console.log(item)}
  items={[
    {text: 'Show comments', key: 'comments', selected: true},
    ActionMenu.Divider,
    {text: 'View file', key: 'view'},
    {text: 'Edit file', key: 'edit'},
    {text: 'Delete file', variant: 'danger', key: 'delete'},
    ActionMenu.Divider,
    {text: 'Open in desktop', key: 'desktop'}
  ]}
/>
```

### Example with leading and trailing visuals

Leading visuals are optional and appear at the start of an item. They can be octicons, avatars, and other custom visuals that fit a small area.

When listing system sections, features, or options, use leading visuals to improve the items' scannability. In user-generated objects, they can help to indicate the item's content type and status.

Trailing visual and trailing text can display auxiliary information. They're placed at the right of the item, and can denote status, keyboard shortcuts, or be used to set expectations about what the action does.

Note these side visuals don't have dedicated interaction targets.

```jsx live
<ActionMenu
  anchorContent="Menu"
  onAction={item => console.log(item)}
  overlayProps={{width: 'medium'}}
  items={[
    {
      text: 'Filter by Values',
      leadingVisual: FilterIcon,
      trailingIcon: () => <span>⌘ + F</span>,
      key: 'filter'
    },
    ActionMenu.Divider,
    {text: 'Sort ascending', leadingVisual: ArrowDownIcon, key: 'sort-asc'},
    {text: 'Sort descending', leadingVisual: ArrowUpIcon, key: 'sort-desc'},
    ActionMenu.Divider,
    {text: 'Hide field', leadingVisual: EyeClosedIcon, key: 'hide'},
    {text: 'Delete field', leadingVisual: TrashIcon, variant: 'danger', key: 'delete'}
  ]}
/>
```

## Best practices

### Leading visual

Leading visuals are optional and appear at the start of an item. They can be octicons, avatars, and other custom visuals that fit a small area.

When listing system sections, features, or options, use leading visuals to improve the items' scannability. In user-generated objects, they can help to indicate the item's content type and status.

Depending on the context, displaying a leading visual may not be necessary. For example, a list of branches in a select panel may not need repeated icons if the surrounding UI provides enough hints about its content type.

### Selection states

Action list items can be selected. Single selections are represented with a check octicon, while multiple selections are represented with a checkbox component. These selection visuals are always placed at the beginning of the item.

When listing selectable items alongside non-selectable items in a menu, use dividers to differentiate between the item types.

Don't mix different types of selections in the same list.

## Accessibility

[Link to the component's WAI-ARIA spec here](https://www.w3.org/TR/wai-aria-practices-1.2)

| Name      | Type                                 |
| :-------- | :----------------------------------- |
| ArrowDown | Moves focus to the next item.        |
| ArrowUp   | Moves focus to the previous item.    |
| Enter     | Calls the action of the focused item |
| Space     | Calls the action of the focused item |
| Escape    | Closes menu.                         |

After closing menu, focus is returned to the `Anchor`
